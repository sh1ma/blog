# sh1ma 文体ルール

このリポジトリのブログ記事 (`src/markdown/posts/*.md`) に共通する著者 sh1ma の文体の癖と、校正時に検出すべき揺れの一覧。校正スキルの判断基準として使う。

---

## 【守るべき癖】(直してはいけない)

以下は「揺れているようで実は sh1ma のスタイル」なので、指摘対象から外す。無理に直すと個性が消える。

### 一人称

- 原則 **「自分」** を使う。
- 「私」「僕」は基本使わない。「私」が混入している場合は「自分」への統一を提案してよい。
- 例外的に「筆者」を使う記事もある (新しい記事の実験)。記事内で統一されていれば残す。

### 口調のミックス

- 1 記事の中で「です・ます」「だ・である」「独り言」が混ざるのは味。
- 例: 手順パートは丁寧語、地の説明や所感は普通体・砕けた口調。
- **段落単位で口調が飛んでいたら指摘してもよい**が、記事全体で強制的に統一しない。
- 「〜っす」「〜だし」「マジで」「めっちゃ」「なんか」も、そういう記事なら残す。

### 語尾・曖昧語

- 「〜っぽい」「〜な気がする」「〜のはず」「〜と思う」「〜みたい」の多用は sh1ma の断言回避癖。消さない。
- 「〜のだ」構文はエッセイでよく出る。
- 「〜ですいません」「〜すいませんでした」の謝りグセ、自虐、自免責は残す。

### 記号・括弧

- **三点リーダーは「・・・」** (半角ドット 3 つ)。全角「…」に置換しない。長い「・・・・・・」も意図的。
- 括弧内の自ツッコミ (「(？)」「(未検証)」「(書くのがめんどくなってしまった)」等) は残す。
- 感嘆符・疑問符の半角/全角混在は、記事全体の傾向に合わせて判断。日本語文中は全角優勢。

### 改行・段落

- **行末 2 スペースのソフト改行**は意図した `<br>` なので削除しない。
- 箇条書きは `-`。`*` に統一しない。

### 結び

- 「またやったら書きます」「気づいたら〜てた」「では！」「以上です。」「〜書けたらいいなと思っています」等の**自免責 + 次回予告** の締めは sh1ma の定型。消さない。

### カタカナ長音

- **技術用語のカタカナは長音省略がデフォルト**: 「サーバ」「ユーザ」「ドライバ」「プログラマ」「エディタ」「ライブラリ」「カテゴリ」「ディレクトリ」「メモリ」。
- 「サーバー」「ユーザー」に統一しない。ただし記事内で「サーバ」と「サーバー」が混在していたら、「サーバ」に寄せる提案は OK。
- 「ハードウェア」「ソフトウェア」「コンピュータ」は長音あり傾向。

### ブランド名の癖

- 「Github」表記は sh1ma に一貫している癖。正式には「GitHub」だが、記事の癖として残す判断もあり。ユーザに寄せる方向を確認する。

### 見出し

- H1 は使わず、Front matter の `title` に任せる。
- H2 (`##`) と H3 (`###`) が主軸。「## まとめ」がほぼ全記事に存在する。
- 番号付き見出しは「### 1. XXX」形式 (半角数字 + ピリオド + 半角スペース)。
- 見出しの日本語は長め (20 文字前後も許容)。

## 【検出すべき揺れ】(指摘対象)

以下は「著者本人も直したほうがよいと思う可能性が高い」揺れ。校正時に拾って寄せる方向とセットで提案する。

### 1. ブランド名の大文字/小文字ブレ

同一記事内で表記が混在する場合は寄せる。

- `Cloudflare` / `cloudflare`
- `Discord` / `discord`
- `GitHub` / `Github` (sh1ma は Github が癖)
- `Xcode` / `XCode`
- `macOS` / `MacOS` (sh1ma は MacOS が癖)
- `OAuth` / `Oauth` / `oauth`
- `JavaScript` / `javascript`
- `Node.js` / `nodejs`

いずれも「記事内での混在」が主な指摘対象。sh1ma の癖 (Github, MacOS) は残す方針でもよいが、混在は必ず片方に寄せる。

### 2. カタカナ長音の同一記事内混在

- 「サーバ」/「サーバー」
- 「ユーザ」/「ユーザー」

同一記事内に両方あれば揺れとして指摘。原則「長音なし」に寄せる。

### 3. 括弧の全角/半角の混在

- 半角 `()` と全角 `（）` が同一記事内で混ざっている場合、記事内で優勢な方に寄せる提案を出す。
- コードブロック内は当然半角のまま。

### 4. バッククォート識別子の前後の半角スペース

- 新記事は「〜 `mise` の設定を〜」のようにスペースを入れる傾向。
- 旧記事は「〜`fs/promises`の`readdir()`〜」とベタ書き。
- **同一記事内で混在**していたら、記事全体の傾向に合わせて寄せる。

### 5. 数字の全角混入

- 半角数字がデフォルト。全角数字 (「９月」「１０」等) が混じっていたら指摘。

### 6. 「〜ヶ月」/「〜か月」/「〜ヵ月」

- 「一ヶ月」(大きな「ヶ」) が優勢。同一記事内混在があれば寄せる。

### 7. 「良い」/「よい」/「いい」

- 段落内で近接して揺れている場合のみ寄せる。記事全体では気にしない。

### 8. 見出し関連

- **番号見出しの `.` / `,` ミス**: `### 2,` は `### 2.` の typo。
- **H2 と H3 の階層飛ばし**: `## foo` の下でセクション内小見出しなのに `## bar` と同階層になっている、など。
- Front matter の `title:` の**先頭空白 (インデント) の有無**、`title` のクォート有無 (`title: "xxx"` / `title: xxx`) は記事によって流儀が違うので単体では指摘しない。ただし同ファイル内で不整合があれば指摘。

### 9. 箇条書きの句点

- 箇条書き各項目末に句点を付けるか付けないか、同一記事内で混在していたら寄せる。原則「付けない」が多数派。

### 10. 全角スペースの混入

- 「Discord　Developer」「Base64　→」「さて、　毎年」のような全角スペースは 100% typo。無条件に指摘。

### 11. em dash `—` の混入

- 新記事だけで使われる文字。旧記事に混入していたら typo か流入と判断してよいが、記事内で意図的に使われていれば残す。

### 12. 画像 alt

- `![]()` のように alt が空の画像があれば alt を書くよう促す。

### 13. Front matter フィールド

- `title` は必須。`publishedAt` は `"YYYY-MM-DD"` 形式でダブルクォート統一。
- `description` と `tags` は新しい記事で使われている。無くても記事は成立するが、あれば揺れとして指摘するのではなく「追加するとよいかも」レベルで注記。

## 【誤字・脱字パターン】

以下は過去の記事で実際に発生したパターン。同種を見つけたら指摘する (例なので網羅ではない)。

- **助詞抜け**: 「実際のコマンド `〜` を逆コンパイルした」 (`で` 抜け)、「1000引き」 (`円` 抜け)、「それなり辛かった」 (`に` 抜け)。
- **助詞重複**: 「公式の手順にに従えば」、「ことをに危機感」、「山をトンネルを伝って」。
- **変換ミス**: 「エラーが**置き**ました」→「起きました」、「Debain」→「Debian」。
- **文末の欠け**: 「セットアップしていきす」→「していきます」。
- **助数詞/単位抜け**: 「1000引き」→「1000円引き」。
- **見出しの `,` / `.` ミス**: 「### 2, XXX」→「### 2. XXX」。
- **変数名 typo**: 記事内で定義した変数と別の綴りで参照している (`postFilePaths` vs `postFilePath`) 場合、コード誤りとして指摘。
- **余剰記号**: 「）」1 個余り、「\_\_」エスケープ残骸、「!!」誤入力など。
- **カッコの閉じ忘れ / 開け忘れ**。

## 【伝わりにくい表現】(要相談カテゴリ)

以下は「校正で直すべきか著者の判断が要る」ゾーン。まず指摘だけ出して、直すかはユーザに委ねる。

- **主語省略で誰の話か戻らないと分からない箇所**。特に段落先頭の一文の主語が離れている場合。
- **指示語 (「これ」「この」「上記」「先の」)** の指す範囲が数段落離れている場合。可能なら明示的に補う提案。
- **前提知識がスキップされて話が飛ぶ箇所**。1 文補うだけで通じる場合が多い。
- **一文が 3 行以上ある長文**。読点で 2 文に割る提案が候補。
- **「〜と思う」の連発**で断定が完全に消えている段落。1〜2 個を「〜だ」「〜になる」に変える提案。
- **括弧内に情報を詰めすぎている行**。長い注釈は本文に格上げ or 削る提案。

**判断が要るところは「直します」ではなく「こういう選択肢があります」の提示で止める**。

## 【校正時の出力フォーマット参考】

指摘は次の形で出すと sh1ma が取捨選択しやすい。

```
## 誤字・脱字
- `src/markdown/posts/xxx.md:12` 「エラーが置きました」→「起きました」の変換ミス
- `src/markdown/posts/xxx.md:47` 「1000引き」→「1000円引き」(「円」抜け)

## 表記揺れ
- `Cloudflare` と `cloudflare` が同一記事内で混在 (L6, L19, L98)。→ `Cloudflare` に寄せるのを推奨
- 「サーバ」と「サーバー」が混在 (L12, L25, L182)。→ sh1ma の他記事に合わせて「サーバ」に寄せるのを推奨

## 構造
- L22 の見出し「### 2, 画面下部の…」→「### 2.」(カンマ→ピリオド)

## 伝わりにくい (要相談)
- L52 「なのでこのケースでは〜」の「このケース」が何を指すか、直前 3 段落戻らないと分からない。「〜のケースでは」と補う?
- L110 の一文が 4 行にわたっている。読点で 2 文に割る?

修正するものを指定してください (例: "全部" / "1, 3, 5 だけ" / "誤字だけ")。
```
