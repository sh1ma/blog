# Commit Skill

変更内容を分析し、適切なコミットメッセージを生成してコミットを実行する。

## 実行フロー

1. **変更内容の収集**
   - `git status` で未追跡ファイルと変更ファイルを確認
   - `git diff --staged` でステージ済みの変更を確認
   - `git diff` でステージされていない変更を確認

2. **変更の分析**
   - 変更されたファイルの種類を特定（機能追加、バグ修正、ドキュメント、リファクタリング等）
   - 変更の影響範囲を把握

3. **コミットメッセージの生成**
   - Conventional Commits 形式に従う
   - プレフィックス: `feat:`, `fix:`, `docs:`, `refactor:`, `chore:`, `style:`, `test:`
   - **件名・本文ともに日本語で記述する**

4. **ユーザー確認**
   - 生成したコミットメッセージを提示
   - 承認を得てからコミット実行

5. **コミット実行**
   - 必要に応じて `git add` でステージング
   - `git commit` を実行
   - `Co-Authored-By` は追加しない（ユーザーの個人ブログのため）

## コミットメッセージのフォーマット

```
<type>: <subject>

<body（任意）>
```

### Type の選択基準

| Type     | 使用場面                         |
| -------- | -------------------------------- |
| feat     | 新機能の追加                     |
| fix      | バグ修正                         |
| docs     | ドキュメントのみの変更           |
| refactor | 機能変更を伴わないコード改善     |
| style    | フォーマット、セミコロン等の変更 |
| test     | テストの追加・修正               |
| chore    | ビルド、設定ファイル等の変更     |

## 注意事項

- 機密情報（.env、credentials等）が含まれていないか確認
- 大きな変更は複数のコミットに分割することを提案
- `--amend` は使用しない（明示的に要求された場合を除く）
